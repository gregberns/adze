# Init & Doctor Specification

## Overview

The `init` command scans the current machine and generates a YAML config file. The `doctor` command dumps full context for AI agent review. Both commands are read-only — they do not mutate the machine or an existing config.

---

## Init Command

### Purpose

Scan current machine state and generate a config YAML from detected packages, preferences, shell configuration, and system settings.

### Invocation

```
adze init [--config output.yaml]
```

- Default output: stdout
- `--config <file>`: write to the specified file

### Detection Sources (darwin)

| Config Section | Source Command | Output Handling |
|---------------|---------------|-----------------|
| packages.brew | `brew leaves` | One formula name per line |
| packages.cask | `brew list --cask` | One cask name per line |
| defaults | `defaults read <domain>` | Parse plist output for curated domains: NSGlobalDomain, com.apple.dock, com.apple.finder, com.apple.screencapture |
| dock | `dockutil --list` | Parse tab-separated output (name, path) |
| identity | `git config --global --list` | Extract user.name, user.email, github.user |
| shell.default | `dscl . -read /Users/$USER UserShell` | Extract shell path |
| shell.oh_my_zsh | `[ -d "$HOME/.oh-my-zsh" ]` | Presence check; parse $ZSH_THEME and plugins from .zshrc |
| machine.hostname | `scutil --get ComputerName` | String value |
| directories | Scan `~/` for common dev directories | Check for: github, gitlab, projects, code, src |
| platform | Runtime detection | Always `darwin` |

### Detection Sources (ubuntu/debian)

| Config Section | Source Command | Output Handling |
|---------------|---------------|-----------------|
| packages.apt | `apt-mark showmanual` | One package name per line (manually installed only) |
| defaults | `gsettings list-recursively` | Filter to org.gnome.desktop.* schemas |
| identity | `git config --global --list` | Same as darwin |
| shell.default | `echo $SHELL` | Shell path |
| shell.oh_my_zsh | `[ -d "$HOME/.oh-my-zsh" ]` | Same as darwin |
| directories | Scan `~/` | Same as darwin |
| platform | `/etc/os-release` | `ubuntu` or `debian` based on ID field |

### Output Format

The generated output MUST be valid YAML that conforms to the Config Schema spec. Comments MUST be included to explain detected values:

```yaml
# Generated by adze init
# Platform: darwin (macOS 15.4, arm64)
# Date: 2026-04-10

name: "<hostname>"
platform: darwin

# Detected from: git config --global
identity:
  git_name: "<value>"
  git_email: "<value>"

# Detected from: brew leaves (<N> packages)
packages:
  brew:
    - <package>
    ...
```

### Edge Cases

| Situation | Behavior |
|-----------|----------|
| Detection tool not installed (e.g., no dockutil) | Skip section; add comment: `# dock: (dockutil not installed)` |
| Detection command errors | Skip section; add comment with error description |
| Empty results for a section | Include section with empty list: `brew: []` |
| Partial detection failure | Continue with remaining sources; note failures in comments |
| brew not installed | Skip packages.brew and packages.cask sections |
| git not configured | Skip identity section |

The output MUST always be valid, parseable YAML regardless of detection failures.

## Doctor Command

### Purpose

Dump full machine configuration context as a self-contained document suitable for AI agent review.

### Invocation

```
adze doctor [--config file.yaml]
```

Output is always to stdout.

### Output Sections

The output MUST include these sections in order:

#### 1. Config
The full resolved config (after includes are merged), as YAML.

#### 2. Dependency Graph
The resolved step execution order with provides/requires visible:

```
1. xcode-cli-tools [provides: xcode-cli-tools] [requires: —]
2. homebrew [provides: homebrew] [requires: xcode-cli-tools]
3. brew-packages [provides: brew-packages] [requires: homebrew]
...
```

#### 3. Validation Results
Output of the `validate` command — all errors and warnings.

#### 4. Platform Information
```
OS: <os> <version> (<arch>)
Package manager: <manager> <version>
Shell: <shell> <version>
Tool version: <version>
```

#### 5. Available Steps Not In Use
Built-in steps from the step library that are not referenced by the current config:

```
Available but not used:
  rust — Install Rust via rustup [any]
  starship — Install Starship prompt [any]
  ...
```

#### 6. Review Questions
A fixed set of review prompts:

1. Are custom step dependencies declared correctly?
2. Are there built-in steps that should replace custom steps?
3. For platform-specific steps, suggest cross-platform equivalents.
4. Review defaults settings for current OS version compatibility.
5. Identify common tools/configs that might be missing.
6. Check for deprecated packages or formulas.
